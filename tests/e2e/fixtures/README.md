# e2e fixtures

Test data used by the end-to-end test suite and the Docker playground. Each
subdirectory maps to one scenario.

---

## shard_1 / shard_2 / shard_3 / shard_4

**Scenario:** `solace-replay` — replay a filtered subset of trades from an archive.

Each shard directory contains a single file:

```
log.rdat.out
```

This is the rdat archive that `bin/playground` tars up and uploads to the target host
as `trading-data-<date>-001200.tar.gz`. The playbook unpacks it into `txn-log/`, then
runs `replay-stub.jar` against the unpacked file.

### log.rdat.out format

One record ID per line. `replay-stub` prints `Replayed <id>` for IDs in the filter
list, `Skipped <id>` for everything else. The e2e tests assert on these lines.

```
trade_001
other_101
other_102
other_103
trade_002
other_104
...
```

All four shards carry the same data (two trade IDs among noise). This is intentional
— the tests verify that the filter is applied correctly regardless of which shard is
targeted, not that the shards have different data.

### Setup (done by playground / bbStruct.sh)

```bash
bb_create_shard_layout trading-host1 /opt/trading shard_1
bb_create_binaries     trading-host1 /opt/trading shard_1
bb_create_archive      trading-host1 /opt/trading shard_1 20250522 tests/e2e/fixtures/shard_1
```

`bb_create_archive` tars the whole fixture directory, uploads it to
`data/trading/archive/trading-data-20250522-001200.tar.gz`, and writes an `.md5`
alongside. The playbook's `tar xvf` step unpacks to `txn-log/`.

## debug-replay

**Scenario:** `debug-replay` — replay a transaction log to rebuild shard state, break
on a specific order, hand off to IntelliJ.

```
debug-replay/
  trading.rdat.in    # inbound events: static data, orders, trades
  trading.rdat.out   # outbound execution reports (not read by bluebird-stub directly)
```

### trading.rdat.in

The full event sequence that `bluebird-stub` reads during startup:

```
product_001 StaticData type=EQUITY exchange=LSE name=Vodafone
product_002 StaticData type=EQUITY exchange=LSE name=Barclays
product_003 StaticData type=EQUITY exchange=LSE name=HSBC
product_004 StaticData type=EQUITY exchange=NYSE name=Apple
product_005 StaticData type=EQUITY exchange=NYSE name=Tesla
product_006 StaticData type=EQUITY exchange=NYSE name=Nvidia
product_007 StaticData type=FUTURES exchange=CME name=ES_Dec26
product_008 StaticData type=FUTURES exchange=CME name=NQ_Dec26
ORD-00001 NewOrderSingle side=BUY qty=100 price=99.50 instrument=product_001
ORD-00002 NewOrderSingle side=SELL qty=50 price=100.00 instrument=product_001
ORD-00003 NewOrderSingle side=BUY qty=200 price=45.25 instrument=product_002
ORD-12345 NewOrderSingle side=BUY qty=500 price=150.00 instrument=product_005
ORD-00004 NewOrderSingle side=SELL qty=75 price=151.00 instrument=product_005
TRD-00001 ExecutionReport orderId=ORD-12345 status=PARTIAL_FILL fillQty=200
ORD-00005 NewOrderSingle side=BUY qty=300 price=200.00 instrument=product_006
TRD-00002 ExecutionReport orderId=ORD-12345 status=FILLED fillQty=300
```

**Phase 1 — Static data (lines 1-8):** Products loaded into `StaticDataHandler`.
Each record sleeps 500 ms to simulate real load time.

**Phase 2 — Recovery (lines 9-17):** Orders dispatched to `OrderEventHandler.process()`.
`ORD-12345` appears at line 12 — this is the target order. `jdi-attacher --condition
orderId=ORD-12345` will fire when this line is replayed.

**Why ORD-12345 is interesting:** It is partially filled (TRD-00001) and then fully
filled (TRD-00002). A developer investigating a double-fill or state inconsistency
would break here to inspect `orderId` and `payload` as the event is processed.

### trading.rdat.out

Outbound execution reports generated by the stub for prior orders. Not read by
`bluebird-stub` itself — present for completeness (the real app would write these;
the stub only reads `.rdat.in`).

### Setup (done by playground / bbStruct.sh)

```bash
bb_create_shard_layout   trading-host1 /opt/trading shard_1
bb_create_debug_binaries trading-host1 /opt/trading shard_1 \
                         tools/bluebird-stub/bluebird-stub.jar 5005
bb_create_rdat           trading-host1 /opt/trading shard_1 \
                         tests/e2e/fixtures/debug-replay/trading.rdat.in \
                         tests/e2e/fixtures/debug-replay/trading.rdat.out
```

`bb_create_debug_binaries` uploads `bluebird-stub.jar` to `app/trading/lib/` and
writes a `start` script that launches the JVM with `suspend=y` JDWP on port 5005.

`bb_create_rdat` copies both files directly into `data/trading/txn-log/` (no archive
step — bluebird-stub reads them live, not from an unpacked tar).

## Adding a new fixture

1. Create a directory under `tests/e2e/fixtures/<scenario-name>/`.
2. Add data files in whatever format the target tool expects.
3. Add a setup call in `bin/playground` using the appropriate `bb_*` function from
   `tests/lib/bbStruct.sh`.
4. Document the fixture here.
